# Copilot instructions

## Build, test, and lint commands

- **Go microservices & shared libraries:** run `go test ./...` inside any service or common library to exercise its unit tests, and narrow a single test with `go test ./path -run TestName`. `go vet ./...` is the default sanity check before shipping a change, and `go build ./cmd/<service>` when you want a compiled artifact.
- **React/Vite frontends (e.g., `blink`, `blinkcheck`, `light-onboarding`, `payment`, `partneradmin`, `merchants`, `smidjan`, `standalonepage`):** run `npm install` (or `yarn install`), then `npm run dev` for local work, `npm run build` for production bundles, and `npm run lint` before merging. Use `npm run preview` to verify the built artifact. These `package.json` files expose only lint/dev/build/preview scripts today—there is no `test` script to target unless a package adds one later.
- **Mobile app (`blikk-app`):** follow `blikk-app/README.md` to populate the `.env` file (e.g., `EXPO_PUBLIC_BLIKK_ENDPOINT`, `EXPO_PUBLIC_APP_ENV`, `EXPO_PUBLIC_ONESIGNAL_APP_ID`) and ensure you have Expo + Apple dev access. Scripts include `npm run start` (`expo start -c`), `npm run dev` (exposes `APP_VARIANT=development`), `npm run stage|qa|prod`, `npm run ios`, and `npm run android`. `npm run lint`, `npm run lint-and-fix`, and `npm run ts:check` cover static checks; `npm run prettier-format` enforces formatting. Tests run via `npm run test` (Jest); target an individual test with `npx jest path/to/test -t "TestName"`.
- **Other toolchains:** execute `devspace dev` in services that include `devspace.yaml` when you need a full Dapr + Kubernetes dev stack, and consult each service’s Dockerfile for containerized builds.

## High-level architecture

- The repo is grouped in `documentation/Repository-Overview.md` into infrastructure libraries (`blikk-engine`, `blikk-runtime`, `jwt`, `logger`, `microservice-template`, `gateway-template`), mobile apps (`blikk-app`, `apploginpage`), React/Vite frontends (Blink, BlinkCheck, merchants, partneradmin, payment, smidjan, standalonepage, and more), API gateways (appapi, appdeeplinkingapi, babapi, blinkapi, blinkcheckapi, ecom, merchantsapi, onboardingapi, partneradminapi, paymentapi, pos, smidjanapi, standaloneposapi, yomapi) and dozens of Go microservices + PSD2 bank integrations that speak through Dapr with MongoDB/Azure backends. Demo/tooling repos (cbev, demoembed, demopos, woo-payment-plugin) live alongside them for experimentation.
- `blikk-engine` orchestrates every service, manages Dapr components across dev/stage/qa/prod, and wires monitoring (Grafana Cloud) plus Sentry for frontend errors; its README documents the access-control policy that denies `/pubsub/**` from the `nginx-ingress` app, so Copilot should not suggest hitting pub/sub endpoints directly without going through Dapr.
- Shared Go helpers include `jwt`, `logger/v2`, and the standard `slog` logger for HTTP clients, structured logging (OpenTelemetry/OTLP), JWT handling, and error translation; avoid pulling in `blikk-runtime` directly. `microservice-template` and `gateway-template` set up idiomatic Dapr pub/sub handlers, OpenAPI (`huma/v2` or `swaggest`), and router wiring (`go-chi/chi`, `gorilla/mux`).
- Core backends (paymentsms, appms, merchantsms, partnersms, notificationms, transactionsms, walletteyams, walletverifonems, etc.) rely on Dapr service invocation + pub/sub, MongoDB, scheduled jobs (e.g., appms credential reminders), and OpenTelemetry instrumentation. PSD2 integrations (psd2*) layer on top of `huma/v2`, `go-chi/chi`, `jwt`, and `dapr/go-sdk` to interact with bank APIs while reusing the same runtime libs.
- Frontends/portal apps are React + Vite (React 18/19) that standardize on `SuperTokens` for auth, `TanStack Query/Router/Table`, Tailwind/Chakra/Radix UI for styling, Axios or fetch for API calls, and Faro/Faro tracing for observability. They render content consumed by merchants and partners while hitting the API gateways, which in turn call the Go services.
- `blikk-app` is a React Native/Expo client that powers payments, NFC, biometrics, and push notifications; it syncs with the same HTTP APIs (SuperTokens, OneSignal, expo-router) and ships alongside the web stack.

## Key conventions

- Every Go service should rely on `slog`, `net/http`, and the shared `jwt` helpers for logging, HTTP wiring, and auth so telemetry and error handling stay consistent—do not import `blikk-runtime`. New backends should be scaffolded from `microservice-template` or `gateway-template` so the Dapr pub/sub wiring, OpenAPI docs, and tracing setup look the same.
- Dapr is the canonical way to invoke peers and subscribe to payment/token events; rely on the Dapr sidecar rather than calling other services directly, and respect the access-control policy (`blikk-engine/apps/*/dapr/components/default-dapr.config.yaml`) that keeps `/pubsub/**` guarded.
- MongoDB (and occasionally Azure Cosmos/Blob Storage) stores most data, so services tend to use the `mongo-driver` plus migration tooling (`golang-migrate`, `paymentsms` data migrations). OpenTelemetry is enabled in every service, and `devspace.yaml` files expose the same components locally.
- React frontends share design/auth primitives: `SuperTokens` for auth, `@tanstack/react-query` for data, Tailwind or Chakra + Radix UI for styling, and `Vite` for bundling. Linting is handled by ESLint/TypeScript; add new lint rules via the `eslint` configs already committed to each frontend package.
- `blikk-app` uses Expo’s environment variables (`EXPO_PUBLIC_*`, `APP_VARIANT`) plus `expo-router`/React Native modules (OneSignal, NFC, Vision Camera). Follow the README’s setup steps (EAS login, `.env`, `pod install` for iOS) before running `npm run ios`/`npm run android`.
- When you need service-level documentation, start with `documentation/Repository-Overview.md` for the big picture, then consult the relevant `README.md` (e.g., `blikk-app/README.md`, `blikk-engine/readme.md`) for deeper context.
